# Database

Wharf uses PostgreSQL with [Drizzle ORM](https://orm.drizzle.team). The
schema lives in a single file, `src/lib/db/schema.ts`, so the entire data
model is readable top to bottom without hunting through a directory of
partial migrations.

## Schema

| Table           | Purpose                                                                                                                      |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `organizations` | The tenant boundary. Has a unique `slug` (used in URLs and cookies) and a unique `clerk_org_id`.                             |
| `memberships`   | Joins a Clerk user id to an organization with a `role` (`owner`, `admin`, `member`). Unique on `(organization_id, user_id)`. |
| `projects`      | Example domain resource, scoped to an organization. Replace or extend this with your actual product's entities.              |
| `activity_logs` | An append-only audit trail of actions taken within an organization.                                                          |

## Local setup

Any Postgres 14+ instance works. The fastest local option is Docker:

```bash
docker run --name wharf-db \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=wharf \
  -p 5432:5432 \
  -d postgres:16
```

Then set in `.env.local`:

```
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/wharf
```

## Migrations

Migrations are plain SQL files generated by Drizzle Kit and committed to
`drizzle/`. The initial migration (`drizzle/0000_init.sql`) creates all four
tables above.

```bash
# After changing src/lib/db/schema.ts, generate a new migration:
npm run db:generate

# Apply all pending migrations:
npm run db:migrate

# Inspect and edit your database visually:
npm run db:studio
```

`npm run db:migrate` runs `src/lib/db/migrate.ts`, a standalone script (not
part of the Next.js server) so it can run in CI or a deploy hook before the
app boots.

## Seeding

```bash
npm run db:seed
```

This creates a demo organization ("Acme Harbor") with three example
projects. By default the membership is attached to a placeholder user id;
set `SEED_CLERK_USER_ID` in `.env.local` to your own Clerk user id (visible
in the Clerk dashboard under **Users**) so the seeded organization actually
shows up when you sign in locally.

## Connection pooling in production

`src/lib/db/index.ts` maintains a single module-level `postgres()` client,
sized to 10 connections in production and 5 in development. If you deploy
to a serverless platform with many concurrent function instances, put a
connection pooler (e.g., PgBouncer, or your provider's built-in pooling —
Neon and Supabase both offer one) in front of Postgres rather than raising
this number, since each serverless instance gets its own pool.
